# 39. WhatsApp-бот: аудит и универсализация под любые услуги

## Цель
Убрать beauty-specific жесткие допущения, чтобы бот корректно работал для любого каталога услуг.

## Scope
- Промпты, intents, системные тексты, fallback-логика.
- Нейтральные термины: service/specialist/slot/booking.
- Tenant-уровневая терминология в отображении.

## Этапы реализации
1. Аудит:
- Найти хардкод под «салон красоты» в bot flows.
- Зафиксировать проблемные места и приоритеты.

2. Generalization:
- Перевести тексты и intent-эвристики на нейтральную модель.
- Добавить конфиг терминов на tenant уровне.

3. Flow integrity:
- Проверить полный booking-flow для разных доменов услуг.
- Проверить edge-cases: перенос, отмена, уточнение услуги/специалиста.

4. QA:
- Smoke-сценарии на 3-4 разных типа бизнеса.

## Риски
- Регресс в текущем стабильном booking-flow.
- Снижение точности intent parsing.

## Критерии готовности
- Бот не зависит от beauty-лексики.
- Один и тот же engine работает для разных типов услуг без правок кода.

## Статус реализации (2026-03-22)
1. Основная реализация завершена:
- tenant-aware terminology и flow config внедрены в bot runtime;
- AI и deterministic ветки приведены к domain-agnostic текстам;
- аудит зафиксирован в `docs/domain-agnostic/audit-findings.md`.
2. Проверки:
- typecheck по bot/api/web — OK.
3. Внешний шаг:
- финальный WhatsApp UAT по реальным номерам выполняется отдельно от кодовой реализации.
