# План переноса дизайна в текущий проект (Next.js)

## Контекст и решения
- Цель: перенести дизайн из архива `Дизайн сайта.zip` в текущий проект без смены стека на Vite.
- Техническая база: текущий `apps/web` на Next.js (App Router + BFF).
- Критерий визуального совпадения: **95%**.
- Стратегия релиза: **целиком** (без поэтапного включения через feature flags).
- Safari/iOS: **обязательный блокер релиза**.

## Этап 00: Аудит и baseline
- Зафиксировать текущие экраны проекта (desktop/mobile) и референс-экраны из архива.
- Сопоставить карту страниц:
  - `Landing`, `Auth`, `Public Booking`, `Booking Success`, `Admin:*`, `404`.
- Сформировать чеклист расхождений `макет vs текущий UI` с приоритетами `P0/P1/P2`.
- Зафиксировать метрику приемки по визуалу: порог 95%.

## Этап 01: Foundations (дизайн-токены)
- Ввести единый слой токенов:
  - цвета (background/surface/text/border/status),
  - радиусы, тени, spacing, typography scale.
- Удалить конфликтующие значения и разрозненные inline-стили.
- Подготовить базовые utility-классы для контейнеров, карточек, форм, таблиц.
- Обеспечить единый вид для public/admin частей.

## Этап 02: Базовые UI-компоненты
- Привести к референсу общие компоненты:
  - `Button`, `Input`, `Select`, `Checkbox`, `Switch`, `Card`, `Tabs`, `Dialog`, `Table`, `Toast`.
- Для каждого компонента закрыть состояния:
  - `default`, `hover`, `focus`, `disabled`, `error`.
- Зафиксировать размеры (`sm/md/lg`) и единые внутренние отступы.
- Избежать дублирования: один набор компонентов для public/admin.

## Этап 03: Layout и навигация
- Реализовать единые каркасы:
  - `PublicLayout`,
  - `AdminLayout`.
- Перенести хедер с language switcher и CTA в переиспользуемый компонент.
- Реализовать sidebar/мобильный хедер/overlay для admin части.
- Проверить sticky/scroll/viewport поведение на мобильных.

## Этап 04: i18n и текстовая консистентность
- Привести все UI-тексты к единой системе ключей (`nav.*`, `auth.*`, `booking.*`, `admin.*`, `common.*`).
- Убрать локальные ad-hoc строки и несогласованные проверки языка.
- Убедиться, что переключение языка применяется сразу на активном экране.
- Добавить fallback-политику для отсутствующих ключей.

## Этап 05: Public screens (UI first, API second)
- Перенести `Landing` 1-в-1 в рамках 95%.
- Перенести `Auth`-экран и связать с текущими BFF/API маршрутами.
- Перенести `Public Booking` step-flow с полноценными состояниями.
- Добавить `Booking Success` и системные состояния (`empty/error/not found`).
- Для форм публичной записи добавить UX-валидации, включая E.164 для телефона.

## Этап 06: Admin screens (по модулям)
- Переносить по модулям:
  - `Dashboard`,
  - `Masters`,
  - `Services`,
  - `Working Hours`,
  - `Exceptions`,
  - `Bookings`,
  - `Settings`,
  - `Notifications`.
- Для каждого модуля:
  - сначала UI + стабильные состояния,
  - затем интеграция реальных данных через текущий API.
- Унифицировать таблицы/фильтры/поиск/empty/loading/error.

## Этап 07: Доступность и UX-полировка
- Контраст, focus-ring, клавиатурная навигация (WCAG AA).
- Микроанимации hover/focus/transition без перегруза.
- Согласованные состояния submit/progress/disabled/error.
- Единые паттерны сообщений об ошибках и успехе.

## Этап 08: Responsive и кроссбраузерное QA
- Обязательные брейкпоинты:
  - `360`, `390`, `768`, `1024`, `1280`, `1440`.
- Обязательная матрица браузеров:
  - Chrome,
  - Firefox,
  - Safari (macOS),
  - iOS Safari,
  - Android Chrome.
- Safari/iOS — блокер релиза: при критических отклонениях релиз запрещен.

## Этап 09: Визуальная регрессия
- Ввести скриншотную проверку ключевых страниц на целевых брейкпоинтах.
- Сравнение `baseline vs candidate` и контроль порога визуального совпадения 95%.
- Регрессии фиксировать до релиза.

## Этап 10: Performance и assets
- Зафиксировать бюджет:
  - LCP/CLS/INP,
  - вес критических стилей,
  - шрифтовая загрузка.
- Определить стратегию шрифтов (preload/fallback).
- Проверить легальность/источники внешних изображений и ассетов.

## Этап 11: Pre-release gate (обязательная приемка)
- Визуальное совпадение >= 95%.
- Пройдены ключевые пользовательские потоки:
  - `Landing -> Auth -> Admin`,
  - `Public Booking -> Success`.
- Пройдены i18n проверки и критические формы.
- Пройдено Safari/iOS QA без блокирующих дефектов.
- Нет критичных UI/API регрессий.

## Этап 12: Rollback план (для цельного релиза)
- Подготовить точку отката на предыдущий стабильный коммит.
- Процедура:
  - откат,
  - деплой,
  - health/ready/smoke проверки.
- Отдельно зафиксировать список hotfix-задач на повторный релиз.

## Definition of Done
- Все целевые экраны перенесены и приняты по визуальному порогу 95%.
- Дизайн-система централизована, inline-хаос устранен.
- Public/Admin используют единые foundations и компоненты.
- i18n работает консистентно и применяется сразу.
- Safari/iOS проверены и соответствуют критериям релиза.
- Документация и чеклисты актуализированы.

## Принципы реализации
- Не менять стек (оставаться на Next.js).
- Не смешивать бизнес-логику и стили в одном слое.
- Переиспользовать компоненты между public/admin.
- Все комментарии в коде приложения писать на английском языке.
- Русский язык использовать только в планах и описаниях.
