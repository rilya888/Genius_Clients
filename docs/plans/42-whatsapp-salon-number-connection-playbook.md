# Этап 42: Подключение нового салона к WhatsApp-боту через Meta

## Цель

Сделать стандартизированный операционный контур, по которому новый салон подключается к WhatsApp-боту по короткой постановке задачи:

- какой салон;
- какой номер телефона;
- какой тариф / план;
- при необходимости: display name / язык / часовой пояс.

После этого исполнитель должен уметь:

1. зайти в Meta;
2. подключить номер к правильному WABA и приложению;
3. получить и проверить технические идентификаторы;
4. обновить проектные переменные и привязки;
5. связать номер с нужным tenant / salon;
6. прогнать end-to-end smoke;
7. зафиксировать результат и откатный путь.

- App: `GeniusClients`

2. В Meta Developers уже включен `Connect on WhatsApp`.

3. Webhook уже настроен на production bot endpoint:

То есть в текущем WABA нет ни одного подключенного номера.

### Вывод из аудита

3. До массового подключения новых салонов нужно считать текущую WhatsApp-конфигурацию переходной и привести её к единому production-standard.

## Целевая архитектура

### Meta side

1. Один Meta App: `GeniusClients`.
2. Один WABA-контур, если бизнес-модель позволяет вести все салоны через один business asset.
3. Несколько WhatsApp phone numbers внутри управляемого контура.
4. Один webhook endpoint на платформу.
5. Входящий маршрут определяется только по `phone_number_id`.

### Project side

1. Один bot service.
2. Один inbound webhook endpoint.
3. Tenant / salon routing только по `phone_number_id`.
4. Отдельный token map по номерам:
- `WA_ACCESS_TOKEN_BY_PHONE_JSON`
5. `WA_ACCESS_TOKEN` используется только как временный fallback, пока не завершена полная миграция.

## Обязательные входные данные для каждой новой задачи

Каждая задача на подключение нового салона должна содержать:

1. Название салона.
2. Tenant slug салона.
3. ID салона / account, если уже создан в системе.
4. Номер телефона в международном формате E.164.
5. Тариф / план.
6. Display name для WhatsApp Business.
7. Страна регистрации номера.
8. Базовый язык салона (`it` / `en` и т.д.).
9. Часовой пояс салона.
10. Контактное лицо, которое получит код верификации номера.
11. Подтверждение, что номер:
- новый;
- не используется как обычный WhatsApp-чат;
- готов к SMS/voice verification.

## Обязательные технические артефакты после подключения

После каждой операции должны быть зафиксированы:

1. `phone_number_id`
2. `display_phone_number`
3. `verified_name`
4. `waba_id`
5. `business_id`
6. token source
7. tenant / account / salon binding
8. дата подключения
9. кто выполнял подключение
10. результат smoke test

## Переменные и секреты, которые участвуют

### Уже существующие ключевые env

- `WA_VERIFY_TOKEN`
- `WA_WEBHOOK_SECRET`
- `META_APP_SECRET`
- `WA_PHONE_NUMBER_ID`
- `WA_ACCESS_TOKEN`
- `WA_ACCESS_TOKEN_BY_PHONE_JSON`
- `WA_ENTERPRISE_ROUTING_REQUIRED`
- `API_URL`
- `BOT_URL`
- `INTERNAL_API_SECRET`

### Production-standard по секретам

Нужно стремиться к такому состоянию:

1. `META_APP_SECRET` хранится только как secret.
2. `WA_VERIFY_TOKEN` хранится только как secret.
3. Все outbound токены живут в `WA_ACCESS_TOKEN_BY_PHONE_JSON`.
4. `WA_ACCESS_TOKEN` после миграции остается только как временный fallback или убирается совсем.
5. В проектной документации токены не хранятся в открытом виде.

## Финальный операционный сценарий подключения нового номера

## Фаза 1. Preflight

Перед подключением нового номера нужно проверить:

1. Салон уже создан в системе.
2. У салона есть корректный tenant slug.
3. Есть admin-owner доступ к салону.
4. Есть согласованный тариф.
5. Есть готовый номер для подключения.
6. Есть доступ к Meta App и WABA.
7. Известно, кто получит verification code.

### Блокирующие условия

Подключение нельзя начинать, если:

1. салон не создан;
2. tenant slug не утвержден;
3. номер уже используется в другом WhatsApp Business контуре;
4. неизвестно, кто подтвердит код;
5. неясно, в какой WABA надо добавлять номер.

## Фаза 2. Meta audit перед конкретным подключением

Перед добавлением каждого нового номера нужно проверить в Meta:

1. правильное приложение открыто (`GeniusClients`);
2. нужный WABA выбран;
3. webhook по-прежнему указывает на production bot URL;
4. `messages` webhook field подписан;
5. есть права на добавление phone number;
6. нет конфликта display name;
7. нет уже существующего номера, который ошибочно считают отключенным.

### Ожидаемый результат фазы

Исполнитель должен заранее знать:

- в какой WABA добавляется номер;
- через какой app он обслуживается;
- будет ли использоваться существующий system token или нужно обновить token map.

## Фаза 3. Добавление номера в Meta

Шаги:

1. Открыть WhatsApp Manager.
2. Выбрать нужный WABA.
3. Нажать `Add phone number`.
4. Заполнить business profile:
- display name;
- category;
- business description при необходимости;
- контактные данные;
- сайт, если нужен.
5. Ввести номер телефона.
6. Выбрать способ верификации:
- SMS;
- voice.
7. Получить код у ответственного лица.
8. Подтвердить код в Meta.
9. Дождаться создания phone asset.

### Что нужно зафиксировать сразу после добавления

1. `phone_number_id`
2. `display_phone_number`
3. `verified_name`
4. `status`
5. `quality_rating`
6. `code_verification_status`

## Фаза 4. Проверка webhook-контура

После добавления номера нужно убедиться, что:

1. приложение не потеряло webhook settings;
2. `messages` по-прежнему subscribed;
3. callback URL не изменился;
4. verify token не был затронут;
5. новый номер реально обслуживается этим же webhook-контуром.

## Фаза 5. Токен и outbound-доступ

### Рекомендуемый стандарт

Для каждого номера должна быть запись в:

`WA_ACCESS_TOKEN_BY_PHONE_JSON`

Даже если фактически используется один и тот же system user token для нескольких номеров.

### Почему так лучше

1. проще ротация;
2. проще аудит;
3. проще точечный rollback;
4. проще проверка token health;
5. не возникает двусмысленности, какой токен обслуживает какой номер.

### Действия

1. Получить актуальный токен, который имеет права на отправку от нового номера.
2. Обновить `WA_ACCESS_TOKEN_BY_PHONE_JSON` в Railway.
3. Добавить запись:

```json
{
  "NEW_PHONE_NUMBER_ID": "TOKEN_FOR_THIS_PHONE"
}
```

4. Не затирать существующие номера.
5. Если токен общий, все равно явно добавить этот `phone_number_id` в JSON map.

## Фаза 6. Project-side binding номера к салону

После Meta надо связать новый номер с нужным tenant / salon.

Нужно иметь или создать binding со следующими полями:

1. `phone_number_id`
2. `waba_id`
3. `business_id`
4. `tenant_id`
5. `account_id`
6. `salon_id`
7. `is_active`
8. `display_name`
9. `connected_at`
10. `connected_by`

### Правила

1. Один `phone_number_id` может принадлежать только одному салону.
2. Один салон на не-enterprise MVP должен иметь один активный номер.
3. Для enterprise дальше можно расширять до нескольких салонов / нескольких номеров, но routing все равно должен оставаться однозначным.

## Фаза 7. Проверка проектных переменных и деплой

После обновления Meta и env нужно:

1. проверить секреты в Railway;
2. обновить:
- `WA_ACCESS_TOKEN_BY_PHONE_JSON`
- при необходимости `WA_PHONE_NUMBER_ID` только если текущий single-number fallback еще используется;
3. удостовериться, что `META_APP_SECRET`, `WA_VERIFY_TOKEN`, `WA_WEBHOOK_SECRET` не затронуты;
4. инициировать redeploy bot-service;
5. проверить health после деплоя.

### Обязательные проверки

1. `/ready` возвращает `200`
2. token health проходит
3. нет `401/403` на Meta Graph API
4. нет ошибок разбора token map

## Фаза 8. Smoke test нового номера

Smoke после подключения обязателен.

### Inbound test

1. Отправить сообщение на новый номер.
2. Проверить webhook receive в логах.
3. Проверить, что извлекается правильный `phone_number_id`.
4. Проверить, что routing указывает на нужный salon.

### Outbound test

1. Бот отвечает на входящее сообщение.
2. Meta не возвращает `401`, `403`, `400` по номеру.
3. В логах нет `unknown tenant`.
4. Сообщение имеет `sent` / `delivered` статус.

### Business-flow test

1. Начать диалог записи.
2. Получить список услуг.
3. Получить слоты.
4. Создать тестовую запись.
5. Проверить, что booking появилась в правильном салоне.
6. При необходимости подтвердить запись из админки.

## Фаза 9. Post-connection фиксация результата

После успешного smoke нужно сохранить операционный результат:

1. Какой салон подключен.
2. Какой номер подключен.
3. Какой `phone_number_id` выдан Meta.
4. Где хранится токен.
5. Какой tenant / salon binding создан.
6. Прошел ли inbound/outbound smoke.
7. Какие ограничения или нюансы остались.

## Rollback-сценарий

Если подключение прошло неуспешно:

1. Удалить новый `phone_number_id` из routing/binding слоя.
2. Откатить `WA_ACCESS_TOKEN_BY_PHONE_JSON`.
3. Перезапустить bot-service.
4. Если проблема на стороне Meta:
- отключить номер;
- не публиковать его как активный;
- зафиксировать причину сбоя.

## Что нужно доработать в проекте до полного production-standard

## 1. Убрать single-number assumptions из admin integrations status

Сейчас статус WhatsApp в admin API завязан на:
- `WA_PHONE_NUMBER_ID`
- `WA_ACCESS_TOKEN`

Это нужно переводить на multi-number aware проверку.

## 2. Сделать явный реестр WhatsApp-номеров

Нужен super-admin / ops слой, где видны:

1. номер;
2. `phone_number_id`;
3. WABA;
4. салон;
5. статус;
6. quality;
7. token health;
8. last inbound / outbound activity.

## 3. Ввести явный runbook “подключить салон к WhatsApp”

Исполнитель не должен каждый раз заново исследовать:
- где в Meta номер;
- какой токен использовать;
- что именно обновлять в проекте.

## 4. Перевести текущую конфигурацию из переходной в консистентную

Поскольку аудит показал, что текущий WABA пустой, нужно отдельным этапом:

1. понять судьбу текущего боевого номера;
2. либо вернуть его в этот WABA;
3. либо пересобрать проектный env под фактический Meta asset;
4. после этого только масштабировать дальше.

## Канонический формат постановки задачи на будущее

Каждая будущая задача на подключение должна выглядеть так:

- салон: `Название салона`
- tenant slug: `tenant-slug`
- salon/account id: `...`
- номер: `+39...`
- display name: `...`
- план: `starter | pro | business | enterprise`
- язык: `it | en`
- часовой пояс: `Europe/Rome`
- контакт для кода: `имя / телефон`

После такой постановки исполнитель должен пройти весь playbook без дополнительных архитектурных решений.

## Definition of Done

Подключение нового салона к WhatsApp-боту считается завершенным, когда одновременно выполнено следующее:

1. номер добавлен в Meta;
2. номер верифицирован;
3. известен `phone_number_id`;
4. webhook-контур активен;
5. token map обновлен;
6. номер привязан к правильному salon / tenant;
7. bot-service задеплоен с актуальными секретами;
8. inbound smoke пройден;
9. outbound smoke пройден;
10. тестовая запись создана и попала в нужный салон;
11. результат зафиксирован в операционной документации.
