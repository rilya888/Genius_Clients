# Этап 01: Project Setup & Infrastructure (детальный план)

## Цель этапа

Сформировать устойчивый фундамент проекта без реализации бизнес-фич: структура монорепо, среда разработки, CI/CD, инфраструктурные договоренности между сервисами и правила, которые не придется ломать на этапах 02-15.

## Принципы на перспективу

- Любая настройка на этапе 01 должна учитывать будущую мультитенантность, поддомены, интеграции WA/TG/Stripe и отдельный bot-сервис.
- Контракты между `web`, `api`, `bot` и `packages/*` фиксируются заранее, чтобы избежать рефакторинга при росте.
- Инфраструктура сразу делится на `development`, `staging`, `production`.
- Никаких “временных” секретов в репозитории.

## Scope этапа

Входит:
- Структура монорепозитория и соглашения по пакетам.
- Базовые tooling-решения (TypeScript strict, линтинг, форматирование, workspace scripts).
- CI-пайплайн качества и сборки.
- Railway-подготовка (проекты/сервисы/переменные/домены) на уровне плана и чеклиста.
- Стандартизированный `.env.example`.
- Базовые operational-конвенции для следующих этапов.

Не входит:
- Реализация продуктовых API, UI, бота, БД-схемы, webhooks.
- Внедрение платежей, уведомлений, интеграций.
- Продвинутая observability и security-реализация (они детализированы в этапах 14-15).

## Целевые артефакты

- Описанный и согласованный layout монорепо.
- Матрица сервисов и их ответственности (`web`, `api`, `bot`, `packages`).
- Список окружений и переменных с назначением.
- CI workflow с обязательными проверками.
- Чеклист запуска нового окружения (staging/prod).
- Decision log по ключевым технологиям (короткие ADR-записи).
- Onboarding checklist для нового разработчика.
- Таблица ownership по артефактам этапа.

## Детальный Work Breakdown

### 1.1 Архитектурные решения (ADR-минимум)

Нужно зафиксировать 5 решений:
- Package manager: `pnpm workspaces`.
- Frontend shell: `Next.js (App Router)`.
- API framework: `Hono` или `Express` (выбрать один и зафиксировать).
- ORM: `Prisma` или `Drizzle` (выбрать один и зафиксировать).
- Хостинг: `Railway` как primary runtime.

Выход:
- Короткие ADR-пункты: “решение / почему / последствия”.

### 1.2 Структура монорепо и границы модулей

Целевая структура:

```
genius_clients/
├── apps/
│   ├── web/
│   ├── api/
│   └── bot/
├── packages/
│   ├── db/
│   ├── i18n/
│   └── shared/
├── docs/
│   └── plans/
└── infra/ (опционально: шаблоны/заметки по Railway)
```

Правила:
- `apps/*` не импортируют друг друга напрямую.
- Общие типы и утилиты только через `packages/shared`.
- Доступ к БД инкапсулируется в `packages/db`.
- Словари и API перевода инкапсулируются в `packages/i18n`.

### 1.3 Контракты между сервисами (без реализации)

Нужно определить:
- Как `web` определяет tenant (по `Host`/subdomain).
- Как `api` принимает tenant-контекст: **BFF.** Браузер → Next.js → api. Tenant из Host на сервере Next.js, в api передаётся доверенным заголовком (X-Internal-Tenant-Id + shared secret).
- Как `bot` связывается с `api` (HTTP, не db) и где хранит conversation state (Redis в prod).
- Общий формат health endpoint для сервисов.

Выход:
- Таблица “кто с кем общается / по какому контракту”.
- Решение: **BFF** — public API не вызывается напрямую из браузера.
- Зафиксированная версия публичного API: префикс `/api/v1` с первого дня.

### 1.4 Стратегия окружений

Окружения:
- `development` (локальная разработка).
- `staging` (интеграционные тесты и QA).
- `production` (боевой контур).

Для каждого окружения зафиксировать:
- Домены.
- Набор сервисов.
- Источник секретов.
- Политику деплоя (ручной/авто).

### 1.5 Переменные окружения и секреты

Сформировать финальный список переменных с группировкой:
- Core: `DATABASE_URL`, `JWT_SECRET`, `REDIS_URL` (staging/prod), `INTERNAL_API_SECRET` (для BFF → api).
- AI: `OPENAI_API_KEY`.
- Payments: `STRIPE_*`.
- Messengers: `WA_*`, `TG_*`.
- Email: `EMAIL_API_KEY`, `EMAIL_FROM`.
- Public client vars: `NEXT_PUBLIC_*`.

Требования:
- В `.env.example` только шаблоны/заглушки.
- Для каждого ключа добавить “где используется”.
- Указать owner ключа (кто выдает/ротирует).
- Отдельно фиксировать, какие ключи только для `staging/prod`.

### 1.6 CI/CD baseline

Минимальный пайплайн на PR/push:
- Установка зависимостей.
- Линтинг.
- Typecheck.
- Build.

Требования к pipeline:
- Fail-fast при ошибке качества.
- Одинаковая версия Node/pnpm локально и в CI.
- Отдельный статус-чек, обязательный для merge.

### 1.7 Railway baseline (подготовка)

Определить:
- 3 сервиса: `web`, `api`, `bot`.
- Подключение PostgreSQL.
- Разделение переменных по `staging`/`production`.
- Поддоменная стратегия: `app.yourapp.com`, `api.yourapp.com`, `*.yourapp.com`.

Проверки:
- Есть чеклист создания нового Railway environment.
- Есть checklist верификации после деплоя.
- Есть минимальный мониторинг расходов (наблюдение без hard-лимитов) по Railway/OpenAI/WA.

### 1.8 Quality gates и coding conventions

Нужно зафиксировать:
- TypeScript strict.
- Стандарты линтинга/форматирования.
- Правило импорта внутренних пакетов.
- Именование файлов, папок, env-переменных.

Важно:
- Комментарии в коде приложения только на английском.
- Русский язык допускается в `docs/plans/*` и описаниях.

### 1.9 Onboarding checklist (без реализации кода)

Нужно описать:
- Какие инструменты нужны локально (Node, pnpm, Docker/psql при необходимости).
- Как проверить корректность окружения.
- Где брать и как заполнять переменные окружения.
- Какие базовые команды проверки использовать перед PR.

### 1.10 Подготовка к этапам 02-06

Этап 01 должен явно подготовить:
- `packages/db` для миграций (этап 02).
- `packages/i18n` для словарей (этап 03).
- `apps/api` каркас роутинга (этап 04).
- `apps/web` каркас public/admin зон (этапы 05-06).

### 1.11 Подготовка к этапам 08-11

Нужно заранее учесть:
- `apps/bot` как отдельный deployable сервис.
- Конфигурацию webhook URL для WA/TG.
- План по scheduler/worker для reminders.

## Зависимости и последовательность

1. Сначала ADR-решения (1.1), затем структура (1.2).
2. После структуры зафиксировать сервисные контракты (1.3).
3. Параллельно можно делать env-матрицу (1.4-1.5) и CI baseline (1.6).
4. Railway baseline (1.7) после фиксации env и сервисов.
5. Завершить quality conventions и readiness к следующим этапам (1.8-1.10).
6. Проверить готовность перехода на этап 02 по DoR.

## Риски этапа и профилактика

- Риск: ранний выбор стека окажется неудобным.  
  Мера: ADR с критериями и явными trade-off.
- Риск: дрейф окружений dev/staging/prod.  
  Мера: единая env-матрица и checklist создания окружений.
- Риск: “скрытые” зависимости между сервисами.  
  Мера: таблица контрактов и запрет cross-import между `apps/*`.
- Риск: проблемы деплоя всплывут поздно.  
  Мера: CI + ранняя проверка Railway baseline.
