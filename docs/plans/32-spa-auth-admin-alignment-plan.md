# 32. План приведения нового SPA фронта в полностью рабочее состояние с backend

## Цель

Обеспечить стабильную и предсказуемую работу нового фронта (`apps/web-vite`) с существующим backend (`apps/api`) без возврата к старому дизайну/старому UI-слою.

## Контекст проблемы

Симптомы, которые наблюдались:
- периодически не работает логин;
- после успешного логина часть данных в админке не отображается;
- поведение меняется после деплоя UI;
- риск рассинхронизации контрактов между новым SPA и API.

Корневой класс проблем:
- несовместимость auth/session flow (refresh/access, CSRF, заголовки);
- неконсистентный API target в production;
- различия в ожиданиях форматов ответов и ошибок;
- неполный набор smoke-gates перед релизом.

## Область работ

- Только WhatsApp-направление (Telegram не входит в объём текущего этапа).
- Новый фронт `web-vite` считается основным клиентом.
- Backend дорабатывается только там, где нужно для стабильного SPA-контракта.

## Архитектурные принципы

- Единый источник истины для auth/session — backend API.
- Контракт важнее реализации: сначала фиксируем API-ожидания, затем код.
- Тенант-границы обязательны на каждом чтении/записи.
- Все изменения должны быть проверяемы автоматическими smoke-сценариями.

## Рабочие потоки

### 1. Auth/session контракт (обязательный)
- Зафиксировать единый формат ответов для:
  - `POST /auth/login`
  - `POST /auth/refresh`
  - `POST /auth/logout`
  - `GET /auth/me`
- Унифицировать хранение/обновление access token на SPA-стороне.
- Ввести строгую обработку `401`:
  - одна попытка refresh;
  - при повторном `401` — принудительный logout и редирект на login.

### 2. CSRF и заголовки
- Для state-changing методов (`POST/PUT/PATCH/DELETE`) гарантировать проброс `x-csrf-token`.
- Исправить объединение headers так, чтобы `Authorization` не терялся при разных форматах заголовков.

### 3. API base URL и окружения
- Убрать неявные зависимости от localhost в production build.
- Ввести детерминированный runtime-resolve API endpoint.
- Добавить smoke-проверку, что production bundle указывает корректный API target.

### 4. Admin API совместимость
- Проверить все критичные страницы админки:
  - services, staff, bookings, schedule, notifications.
- Для каждой страницы: корректная авторизация, обработка ошибок, отображение пустых/частично заполненных данных.

### 5. Наблюдаемость и диагностика
- Унифицировать формат frontend ошибок:
  - статус;
  - message;
  - requestId.
- Добавить runbook инцидента login/admin desync.
- В логах API иметь возможность быстро отследить auth-путь конкретного запроса.

### 6. Релизные ворота (release gates)
- Перед каждым релизом web-vite обязательны:
  - `spa:public` smoke;
  - `spa:auth-admin` smoke;
  - `web-vite api target` smoke.
- Релиз блокируется при падении хотя бы одного gate.

## План внедрения по этапам

### Этап A. Стабилизация auth
- Привести login/refresh/me/logout к единому контракту.
- Привести Protected Route к реальной валидации сессии, а не только к проверке local storage.

### Этап B. Стабилизация admin API
- Внедрить токен-aware клиент с auto-refresh и controlled logout.
- Пройти ключевые сценарии админки и исправить контрактные расхождения.

### Этап C. Стабилизация production-контура
- Нормализовать API base resolver.
- Добавить и зафиксировать smoke-gates.
- Обновить runbook.

### Этап D. Подготовка к масштабированию (enterprise/multi-salon)
- Проверить, что auth/session слой не привязан к single-tenant предположениям в UI-клиенте.
- Зафиксировать требования к tenant switch/tenant scope для будущего multi-salon режима.

## Критерии готовности

- Логин стабильно работает в production и staging.
- После логина админка открывает данные текущего tenant без ручного пере-логина.
- При истечении access token UI автоматически делает refresh без потери контекста.
- Все release gates проходят зелёными перед релизом.
- Есть исполнимый runbook на типовой инцидент "логин есть, админка пустая/ломается".

## Риски и меры

- Риск: частичная совместимость старых и новых endpoint-ожиданий.
  - Мера: compatibility matrix и явные adapter-функции в API client.
- Риск: скрытые ошибки окружения (не тот API URL).
  - Мера: runtime валидация + smoke на target host.
- Риск: регрессии при доработках бота.
  - Мера: разделение зон изменений и обязательный auth-admin smoke после merge.

## Что дальше после завершения этого плана

- Переход к enterprise-функциям multi-salon в рамках подписки:
  - несколько салонов в одной организации;
  - отдельный WhatsApp номер и бот-конфиг на салон;
  - централизованное управление доступами и лимитами по тарифу.
